三、全局统一编码规范（强制落地）
1. 通用编码原则（必须严格遵守）
所有代码开发必须遵循：SOLID原则、DRY原则、KISS原则、高内聚低耦合、模块化设计
强制统一：错误处理、日志规范、配置管理、异常捕获机制
2. 代码硬性要求（缺一不可）
所有产出代码必须满足：可运行、可维护、可扩展、可测试、带边界检查、带异常处理、带日志输出、带类型定义、带接口定义、带安全校验
3. 严格禁止编码行为
禁止魔法值、禁止硬编码、禁止重复代码、禁止全局变量污染、禁止巨型函数、禁止巨型类、禁止无注释复杂逻辑、禁止无异常处理业务代码
4. TS专属规范
禁止泛滥使用any类型，所有数据结构优先使用interface/type定义；统一函数式组件开发，废弃类组件；所有异步请求统一封装，禁止裸写请求逻辑、禁止滥用链式回调
5. 格式统一规范
全局缩进统一、引号统一、语句结尾规范；变量、文件、组件、接口、数据库字段命名全程统一规范化，禁止随意自定义命名规则

四、目录与文件管理规范
1. 严格遵循架构师定义的项目目录结构，禁止私自新建目录、随意调整层级、乱存文件
2. 新增页面、组件、工具类、接口、配置文件，必须存放至对应规范目录，统一归类
3. 公共组件、通用工具、全局配置统一抽离复用，禁止重复编写工具逻辑
4. 所有AI相关能力、Prompt模板、模型配置、Agent逻辑统一归集专属AI目录，单独维护

1. 所有功能开发严格对齐产品PRD，仅围绕[核心业务领域]展开

4. 所有AI智能能力必须贴合业务场景，禁止开发无业务价值的AI冗余逻辑

六、注释与文档规范
1. 所有复杂逻辑、工具函数、AI算法逻辑、核心业务流程、权限校验逻辑，必须添加完整中文注释
2. 重大功能迭代、架构调整、AI能力升级、数据库变更，必须输出简易开发说明与变更记录
3. 项目所有模块必须配套输出：架构图、ER图、API文档、时序图、部署文档、README、配置说明、环境变量说明、测试说明
4. 文档工程师定期统一规整所有文档，保证同步最新代码与业务逻辑

---
七、容错、测试与质量规范
1. 所有代码自带基础容错机制、严格参数校验，空值校验、边界场景校验
2. 所有接口、页面渲染、AI模型调用、文件操作、网络请求，必须完善异常处理，杜绝报错崩溃，白屏、接口卡死
3. 所有功能必须覆盖单元测试、集成测试，AI测试能力需额外覆盖场景化测试、边界测试
4. 上线前必须完成安全审计、性能优化、回归测试，三审通过方可交付

---
八、数据库强制规范
1. 所有数据表必须建立合理索引，避免慢查询、N+1查询问题
2. 业务数据操作必须使用事务，保证数据一致性
3. 所有数据表统一规范命名、统一审计字段、支持软删除、支持乐观锁
4. 所有查询强制使用分页，禁止全表查询大数据量

---
九、安全强制规范（项目红线）
所有开发必须内置全套安全机制，缺一不可：
- JWT身份认证机制
- RBAC精细化权限模型
- SQL注入防护、XSS防护、CSRF防护
- 接口限流、防刷、防重放机制
- 所有敏感数据脱敏处理
- 全链路操作审计日志留存
- 核心数据加密存储、传输
- 全站HTTPS适配

---

4. 严格环境隔离：开发、测试、预发、生产环境完全独立，禁止跨环境操作
