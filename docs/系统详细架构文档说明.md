### 1.2 每个文件夹的用途(大白话解释)

**前端部分 (apps/frontend/)：**
- **api/** - 像是"电话簿",记录了前端怎么联系后端
- **views/** - 用户看到的所有页面(登录页、列表页、详情页等)
- **router/** - 像是"导航仪",告诉系统去哪个页面
- **stores/** - 像是"共享储物柜",存全局数据(当前用户信息等)
- **utils/** - 像是"工具箱",放公共方法(格式化日期、验证等)
- **components/** - 像是"乐高积木",可以重复使用的UI组件

**后端部分 (apps/backend/)：**
- **auth-service/** - "门卫",负责验证用户身份(登录、注册、Token)
- **user-service/** - "人事部",管理用户信息、角色、权限
- **project-service/** - "项目经理办公室",管理测试项目
- **testcase-service/** - "测试用例仓库",管理测试用例的增删改查
- **defect-service/** - "缺陷跟踪器",记录Bug
- **requirement-service/** - "需求柜",存放需求文档
- **dashboard-service/** - "数据统计中心",展示各种统计数据
- **common/** - "公共资料库",所有服务都可以借用的工具类

**数据库部分 (backend/database/)：**
- **schema.sql** - "数据库蓝图",定义了所有表结构
- **data.sql** - "初始数据",插入一些基础数据

## 第二部分：项目启动流程

### 2.1 项目从哪个文件启动

**前端启动：**
- 入口文件：`apps/frontend/src/main.ts`
- 这是Vue3应用的启动点，类似于"按下电脑开机键"

**后端启动：**
- 每个微服务都有自己的启动类，例如：
  - `testcase-service/src/main/java/com/aitest/testcase/TestcaseApplication.java`
- 这是Spring Boot应用的启动点

### 2.2 启动后先加载什么

**后端启动流程(Java Spring Boot)：**

```
启动TestcaseApplication
    ↓
加载配置文件application.yml
    ↓
初始化数据库连接池
    ↓
扫描并注册Controller
    ↓
扫描并注册Service
    ↓
扫描并注册Mapper
    ↓
启动Tomcat服务器
    ↓
监听端口9005
```

**配置文件加载顺序：**
1. `application.yml` - 主配置文件(端口、数据库连接)
2. `pom.xml` - Maven依赖配置
3. 数据库配置 - 连接MySQL

### 2.3 配置文件位置

**后端配置：`apps/backend/testcase-service/src/main/resources/application.yml`**

```yaml
server:
  port: 9005  # 这个服务监听9005端口

spring:
  datasource:
    url: jdbc:mysql://localhost:3306/ai_test_platform  # 数据库地址
    username: root   # 数据库用户名
    password: root   # 数据库密码

mybatis-plus:
  mapper-locations: classpath*:/mapper/**/*.xml  # SQL映射文件位置
```

**前端配置：`apps/frontend/src/utils/request.ts`**
- 配置了后端API地址

### 2.4 数据库连接

**在哪里配置：** `application.yml`

```yaml
spring:
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/ai_test_platform?useUnicode=true&characterEncoding=utf-8
    username: root
    password: root
```

**连接池配置：**
- Spring Boot会自动配置HikariCP连接池
- 默认连接数：10个

### 2.5 路由注册

**前端路由：** `apps/frontend/src/router/index.ts`

```typescript
routes: [
  { path: '/login', component: Login },
  { path: '/testcase', component: Testcase },
  { path: '/project', component: Project },
  // ...
]
```

**后端接口：** `@RequestMapping("/api/v1/testcases")`

在Controller类上使用注解声明路由前缀

### 2.6 程序启动后发生了什么

**完整启动流程：**

1. 用户打开浏览器访问 http://localhost
2. 前端加载 `main.ts`，初始化Vue应用
3. router加载路由配置
4. 显示登录页面
5. 用户输入用户名密码，点击登录
6. 前端调用 `POST /api/v1/auth/login`
7. 后端查询数据库验证用户
8. 返回JWT Token给前端
9. 前端保存Token，跳转到首页
10. 用户点击"测试用例"菜单
11. 前端调用 `GET /api/v1/testcases`
12. 后端查询数据库返回用例列表
13. 前端渲染用例列表页面

### 3.2 核心表结构详解

#### 表1：`testcase` (测试用例表)

```sql
CREATE TABLE `testcase` (
    `id` BIGINT AUTO_INCREMENT,           -- 用例ID(主键)
    `project_id` BIGINT NOT NULL,          -- 属于哪个项目(外键)
    `requirement_id` BIGINT,               -- 关联的需求ID
    `title` VARCHAR(256) NOT NULL,         -- 用例标题(必填)
    `description` TEXT,                   -- 用例描述
    `preconditions` TEXT,                 -- 前置条件
    `priority` VARCHAR(32),               -- 优先级 P0/P1/P2/P3
    `type` VARCHAR(32),                    -- 用例类型:功能/边界/异常/安全
    `status` VARCHAR(32),                 -- 状态:草稿/已评审/已批准
    `test_status` VARCHAR(32),             -- 执行状态:未执行/通过/失败
    `test_module` VARCHAR(128),            -- 所属模块
    `steps` TEXT,                         -- 测试步骤
    `expected_result` TEXT,                -- 预期结果
    `actual_result` TEXT,                 -- 实际结果
    `ai_generated` INT,                   -- 是否AI生成:0否,1是
    `created_by` BIGINT,                   -- 创建人ID
    `updated_by` BIGINT,                  -- 更新人ID
    `created_at` DATETIME,                 -- 创建时间
    `updated_at` DATETIME,                -- 更新时间
    `deleted` TINYINT,                    -- 删除标记:0未删,1已删
);
```

**字段作用说明：**
- `id` - 用例的唯一身份证号
- `title` - 用例的名字，比如"登录功能测试"
- `priority` - 优先级，越高越重要(P0最紧急)
- `type` - 什么类型的测试
- `status` - 用例本身的状态(是否审核过)
- `test_status` - 测试执行的结果(通过了吗)
- `steps` - 具体的操作步骤
- `expected_result` - 期望的结果

#### 表2：`testcase_step` (用例步骤表)

```sql
CREATE TABLE `testcase_step` (
    `id` BIGINT AUTO_INCREMENT,
    `testcase_id` BIGINT NOT NULL,        -- 属于哪个用例
    `step_number` INT NOT NULL,           -- 步骤序号(1,2,3...)
    `action` TEXT NOT NULL,               -- 操作步骤内容
    `expected_result` TEXT NOT NULL,      -- 预期结果
```

**字段作用：**
- `testcase_id` - 关联到主表testcase的id
- `step_number` - 第几步(1、2、3)
- `action` - 这一步要做什么操作
- `expected_result` - 这一步期望的结果

#### 表3：`defect` (缺陷表)

```sql
CREATE TABLE `defect` (
    `id` BIGINT AUTO_INCREMENT,
    `project_id` BIGINT NOT NULL,

    `description` TEXT,

    `status` VARCHAR(32) DEFAULT 'new',       -- new/confirmed/assigned/fixing/fixed/verified/closed
    `type` VARCHAR(32) DEFAULT 'bug',

    `module` VARCHAR(128),
    `steps_to_reproduce` TEXT,
    `expected_result` TEXT,
    `actual_result` TEXT,
    `reporter_id` BIGINT NOT NULL,
    `assignee` BIGINT,
```

## 第四部分：核心模块详细分析

### 以【测试用例管理模块】为例进行详细分析

#### 4.1 模块作用

**这个模块干什么？**

用于**管理测试用例的生命周期**，包括：
- 创建新用例
- 编辑已有用例
- 删除废弃用例
- 查询筛选用例
- 查看用例详情(包括步骤)
- 标记用例执行结果
- 统计用例数量

**一句话总结：** "测试用例的仓库管理员"

#### 4.2 模块涉及的接口

```typescript
// 前端API调用 (apps/frontend/src/api/testcase.ts)

GET    /api/v1/testcases              // 查询用例列表(分页)
GET    /api/v1/testcases/{id}         // 获取用例详情
POST   /api/v1/testcases              // 创建新用例
PUT    /api/v1/testcases/{id}         // 更新用例
DELETE /api/v1/testcases/{id}         // 删除用例
GET    /api/v1/testcases/{id}/steps   // 获取用例步骤
GET    /api/v1/testcases/stats        // 获取用例统计
```

**接口详细说明：**

| 接口 | 用途 | 前端什么时候调用 | 请求参数 | 返回数据 |
|------|------|------------------|---------|---------|
| GET /testcases | 查询列表 | 打开用例页面时 | page, size, keyword, priority, status | {list: [], total: 100} |
| POST /testcases | 创建用例 | 点击"新建用例"按钮 | {title, priority, type, steps...} | 新建的用例对象 |
| PUT /testcases/{id} | 更新用例 | 编辑后保存 | {title, priority...} | 更新后的用例 |
| DELETE /testcases/{id} | 删除用例 | 点击删除按钮 | id | success/error |

#### 4.3 接口调用链

**完整流程：前端 → Controller → Service → Mapper → Database**

```
前端 Vue
    ↓ HTTP请求
Controller层 (接收请求)
    ↓ 调用
Service层 (业务逻辑)
    ↓ 调用
Mapper层 (数据访问)
    ↓ SQL
MySQL数据库
```

**每一步详细说明：**

1. **前端发起请求**
   - 用户点击按钮
   - 调用`getTestcaseList()`函数
   - 发送HTTP请求到后端

2. **Controller接收请求** (`TestcaseController.java`)
   - 接收HTTP请求
   - 参数校验
   - 调用Service层

3. **Service处理业务逻辑** (`TestcaseServiceImpl.java`)
   - 编写核心业务逻辑
   - 数据转换
   - 调用Mapper

4. **Mapper执行SQL** (`TestcaseMapper.java`)
   - MyBatis Plus自动生成SQL
   - 操作数据库

5. **数据库返回结果**
   - 查询返回数据
   - 插入返回主键
   - 更新返回影响行数

#### 4.4 CRUD实现过程

**步骤2：前端调用接口**

#### 8.2.2 request工具解析

```typescript
import axios from 'axios'
import { ElMessage } from 'element-plus'

// 创建axios实例
const request = axios.create({
  baseURL: 'http://localhost:9005',  // 后端地址
  timeout: 10000,                    // 超时时间10秒
  headers: {
    'Content-Type': 'application/json'  // 请求头
  }
})

// 请求拦截器
request.interceptors.request.use(
  (config) => {
    // 在发送请求之前做些什么
    const token = localStorage.getItem('token')  // 从本地获取Token
    if (token) {
      config.headers['Authorization'] = `Bearer ${token}`  // 添加到请求头
    }
    return config
  },
  (error) => {
    return Promise.reject(error)
  }
)

// 响应拦截器
request.interceptors.response.use(
  (response) => {
    // 对响应数据做点什么
    const res = response.data
    
    // 判断业务状态码
    if (res.code !== 200 && res.code !== 0) {
      ElMessage.error(res.message || '请求失败')
      return Promise.reject(new Error(res.message || '请求失败'))
    }
    
    return res
  },
  (error) => {
    // 对响应错误做点什么
    ElMessage.error(error.message || '网络错误')
    return Promise.reject(error)
  }
)

export default request
```

**request工具的作用：**

1. **统一配置** - 所有请求都使用相同的baseURL和超时时间
2. **自动添加Token** - 每个请求自动带上JWT Token
3. **统一错误处理** - 所有错误统一提示
4. **统一响应处理** - 判断业务状态码

### 9.1 前端状态管理概述

#### 9.1.1 什么是状态管理？

**状态管理 = 管理全局数据**

```
┌──────────────────────────────────────────────────────┐
│                    状态管理核心概念                      │
└──────────────────────────────────────────────────────┘

想象一个大型商场的"客户服务中心"：

1. 共享储物柜（状态存储）
   - 存储所有客户都需要访问的数据
   - 例如：当前登录用户信息、权限列表
   
2. 储物柜管理员（Store）
   - 负责管理数据的存取
   - 确保数据一致性和安全性
   
3. 客户办理业务（组件使用）
   - 任何页面都可以访问储物柜
   - 不需要层层传递数据

这就是Vue3的Pinia状态管理！
```

**为什么需要状态管理？**

```
传统方式（props层层传递）：
页面A → 页面B → 页面C → 页面D
         ↓         ↓         ↓
       传递      传递      传递
      用户信息   用户信息   用户信息

问题：
- 数据需要经过很多层传递
- 中间层组件不关心这些数据，但必须传递
- 代码混乱，难以维护

使用状态管理后：
页面A ──┐
页面B ──┼──→ 用户Store（共享储物柜）
页面C ──┤
页面D ──┘

优势：
- 数据只存储一份
- 任何页面都能直接访问
- 代码简洁，易于维护
```

#### 9.1.2 Pinia状态管理

**Pinia是什么？**

Pinia是Vue3官方推荐的状态管理库，类似于Vuex但是更简单。

```
┌─────────────────────────────────────────┐
│              Pinia架构                   │
└─────────────────────────────────────────┘

        ┌─────────────────┐

        └────────┬────────┘

┌───────────────┐    ┌───────────────┐
```

#### 9.2.1 Store结构解析

```typescript
// ========================================
// 第1部分：定义用户信息类型
// ========================================

export interface UserInfo {
  id?: number        // 用户ID
  userId?: number   // 用户ID(兼容)
  username?: string  // 用户名
  email?: string    // 邮箱
  phone?: string    // 手机号
  realName?: string // 真实姓名
  departmentId?: number // 部门ID
  status?: number   // 状态
  roles: string[]   // 角色列表 - 如['ADMIN', 'TESTER']
  permissions: string[]  // 权限列表 - 如['testcase:view', 'defect:create']
  createTime?: string   // 创建时间
  updateTime?: string   // 更新时间
}

// ========================================
// 第2部分：创建Store
// ========================================

/**
 * 定义用户Store
 * 语法：defineStore('store名称', () => { ... })
 * 
 * 使用了Vue3 Composition API风格
 * 优点：更好的TypeScript支持、更灵活的代码组织
 */
export const useUserStore = defineStore('user', () => {
  
  // ========================================
  // 第3部分：定义状态（State）
  // ========================================
  
  /**
   * 用户信息
   * ref()用于定义响应式数据
   * 类似于Vue2中的data()
   */
  const userInfo = ref<UserInfo | null>(null)
  
  /**
   * JWT Token
   * 用于身份验证
   */
  const token = ref<string>('')

  // ========================================
  // 第4部分：定义Actions（修改状态的方法）
  // ========================================
  
  /**
   * 设置用户信息
   * 同时保存到localStorage实现持久化
   */
  const setUserInfo = (info: UserInfo) => {
    userInfo.value = info
    localStorage.setItem('userInfo', JSON.stringify(info))
  }

  /**
   * 设置Token
   * 同时保存到localStorage实现持久化
   */
  const setToken = (newToken: string) => {
    token.value = newToken
    localStorage.setItem('token', newToken)
  }

  /**
   * 获取Token
   * 如果内存中没有，从localStorage读取
   */
  const getToken = () => {
    if (!token.value) {
      token.value = localStorage.getItem('token') || ''
    }
    return token.value
  }

  /**
   * 获取用户信息
   * 如果内存中没有，从localStorage读取
   */
  const getUserInfo = () => {
    if (!userInfo.value) {
      const stored = localStorage.getItem('userInfo')
      if (stored) {
        try {
          userInfo.value = JSON.parse(stored)
        } catch (e) {
          console.error('解析用户信息失败:', e)
        }
      }
    }
    return userInfo.value
  }

  /**
   * 退出登录
   * 清空所有状态和localStorage
   */
  const logout = () => {
    userInfo.value = null
    token.value = ''
    localStorage.removeItem('userInfo')
    localStorage.removeItem('token')
  }

  /**
   * 检查用户是否有指定角色
   */
  const hasRole = (role: string): boolean => {
    if (!userInfo.value) return false
    return userInfo.value.roles?.includes(role) || false
  }

  /**
   * 检查用户是否有指定权限
   * 支持通配符匹配，如 'testcase:*' 匹配所有testcase相关权限
   */
  const hasPermission = (permission: string): boolean => {
    if (!userInfo.value) return false
    const perms = userInfo.value.permissions || []
    
    // 超级管理员拥有所有权限
    if (perms.includes('*') || perms.includes('system:*')) {
      return true
    }
    
    // 精确匹配或通配符匹配
    return perms.some(p => {
      if (p === permission) return true
      if (p.endsWith('*')) {
        return permission.startsWith(p.slice(0, -1))
      }
      return false
    })
  }

  /**
   * 判断是否是管理员
   */
  const isAdmin = (): boolean => {
    if (!userInfo.value) return false
    return userInfo.value.roles?.includes('ADMIN') || hasPermission('system:*') || false
  }

  // ========================================
  // 第5部分：导出状态和方法
  // ========================================
  
  return {
    userInfo,
    token,
    setUserInfo,
    setToken,
    getToken,
    getUserInfo,
    logout,
    hasRole,
    hasPermission,
    isAdmin
  }
})
```

```typescript
/**
 * 路由守卫 - 检查登录状态和权限
 */
router.beforeEach((to, _from, next) => {
  // 404页面直接放行
  if (to.name === 'NotFound') {
    next()
    return
  }
  
  // 获取路由配置的认证要求
  const requiresAuth = to.meta.requiresAuth
  
  // 如果不需要认证，直接放行
  if (!requiresAuth) {
    next()
    return
  }
  
  // 检查Token是否存在
  const token = localStorage.getItem('token')
  if (!token) {
    // 没有Token，跳转到登录页
    next('/login')
    return
  }
  
  // 检查角色权限
  const requiredRoles = to.meta.roles as string[] | undefined
  if (requiredRoles && requiredRoles.length > 0) {
    if (!hasRole(requiredRoles)) {
      ElMessage.error('您没有权限访问该页面')
      next(false)  // 阻止导航
      return
    }
  }
  
  // 检查权限码
  const requiredPermission = to.meta.permission as string | undefined
  if (requiredPermission && !hasPermission(requiredPermission)) {
    ElMessage.error('您没有权限访问该页面')
    next(false)  // 阻止导航
    return
  }
  
  // 所有检查通过，放行
  next()
})
```
